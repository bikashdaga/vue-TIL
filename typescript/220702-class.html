# 타입스크립트 객체지향 프로그래밍

자바스크립트가 함수형 프로그래밍에 더 최적화 되어 있기는 하지만, 객체지향이라는 이론이 프로그래밍 세계에서 빼놓을 수 없기 때문에 ES6에서 결국 클래스 문법을 지원하게 됩니다.

자바스크립트는 특성 상 객체지향 프로그래밍에 한계가 있을 수 있지만, 타입스크립트를 사용하게 되면 더욱 엄격하게 코드를 짤 수 있게 됩니다.

다음은 타입스크립트에서 작성되는 클래스 문법입니다.

class Player{
    constructor(
        private firstName:string,
        private lastName:string,
        public nickName:string
    ){}
}

ES6에서의 클래스 문법또한 생성자 함수 정의부터 시작됩니다. this를 통해 클래스를 통해 생성된 인스턴스 자체를 가리키게 되고 해당 인스턴스에 생성자 내부 필드를 초기화시키는 과정으로 이루어집니다. 위의 코드가 자바스크립트로 컴파일되면 다음과 같습니다.

'use strict';
class Player {
    constructor(firstName, lastName, nickName) {
        this.firstName = firstName;
        this.lastName = lastName;
        this.nickName = nickName;
    }
}

꼭 private으로 필드를 등록할 필요는 없습니다. public firstName:string으로 해도 됩니다.

private과 public이 자바스크립트에 그대로 반영되지는 않는 것처럼 보이지만 타입스크립트 내부적으로 이 둘을 구분하여 보호합니다.

# 추상클래스

추상클래스는 자식 클래스로 상속을 보낼 수는 있지만 자신의 클래스를 통해 인스턴스를 생성할 수는 없습니다.

추상클래스 내에는 추상메서드를 작성할 수 있습니다. 함수의 구현부는 작성하지 않고 콜 시그니쳐만 작성해야 합니다. abstract 키워드와 함께 콜 시그니쳐를 작성합니다.

추상클래스를 상속받는 모든 자식클래스들은 반드시 추상메서드의 구현부를 작성해야합니다. 추상클래스의 추상메서드를 구현하지 않으면 에러가 발생합니다.

일반 객체지향 프로그래밍 언어들 처럼 필드 선언에는 private, public, protected가 사용됩니다. 아래의 User 추상클래스로부터 Player가 상속받아 유저네임을 출력하는 추상메서드를 구현하기 위해서는 name 필드에 대해 private이 아닌 protected로 선언해야합니다.

abstract class User{
    constructor(
        protected firstName: string,
        protected lastName: string,
        public nickname:string
    ){}
    abstract getName():string;
}

class Player extends User{
    getName = () => {
        // protected 선언
        // this로 부모 클래스에 접근
        return `${this.firstName} ${this.lastName}`
    }
}

const me = new Player("경준", "박", "쥰");

console.log(me.getName()); // 경준 박

console.log(me.firstName); // protected필드이기 때문에 에러가 발생

필드에 readonly 적용하기

클래스 생성 시 필드를 public으로 선언하면 외부에서 클래스 인스턴스에 접근하여 기존 값을 변경시킬 수 있는 위험성이 존재합니다. 이때 public readonly field: string과 같이 public 키워드 뒤에 readonly 옵션을 붙이면 읽기 전용으로 필드 변경권한이 바뀝니다.

static method

정적 메서드는 클래스에 대한 인스턴스 생성 없이 클래스 자체에서 호출할 수 있는 메서드입니다.

class Dict {
    static hello() {
        console.log('Hello!');
    }
}

Dict.hello(); // Hello!

# 인터페이스

객체에 대한 타입 설명을 위해 기존에는 type키워드를 붙여 사용했었습니다. 또한 대입 연산자로 변수에 저장하는 형태를 가졌는데 인터페이스를 사용하면 다음과 같이 형태가 달라집니다.

// 기존 타입 설명
type Words = {
    name: string,
};

// 인터페이스 타입 설명
interface Words {
    name: string;
}

type 키워드가 interface로 바뀌고 대입연산자를 사용하지 않습니다.

둘 다 객체 타입을 설명한다는 데에 있어서는 동일한 역할을 하므로 원하는 형태를 골라 사용하면 됩니다.

다만 type 키워드가 객체 이외의 다양한 타입 설명을 위해 범용성을 가집니다.

interface는 오직 객체 타입 설명을 위해서만 사용됩니다.

인터페이스는 다른 인터페이스를 상속받아 사용할 수도 있습니다.

interface User{
    name: string
}

// 상속받은 후 Player 내부 타입정의 X
interface Player extends User{
}

// Player내부에 타입을 정의하지 않았음에도
// User를 상속받았기 때문에 name에 접근 가능
const me : Player {
    name:"Jun"
}

&(=equal)로 `type` 불러오기

인터페이스에서는 타입들을 상속받을 수 있었는데, type 키워드로 선언한 객체 타입도 상속받는 것 같은 효과를 낼 수 있습니다.

// type으로 선언
type User = {
    name: string,
};

// & ampersand 연산자 활용
type Player = User & {};

const me: Player = {
    name: 'Jun',
};

# interface with class

인터페이스는 클래스 형태를 정의할 때에 중요하게 사용됩니다. 자바의 인터페이스와 동일한 역할을 하게 됩니다.

추상메서드를 정의하고 이를 상속받아 메서드 및 필드 구현을 하는 것은 일반적인 방식입니다. 이때 타입스크립트 특성 상 컴파일이 진행됨에 따라 추상클래스의 코드들이 자바스크립트 코드로 컴파일 됩니다. 자바스크립트에서도 클래스 문법이 사용되기 때문입니다.

이에 반해 인터페이스로 클래스의 형태를 정의한 뒤 이를 다른 클래스에서 받아와 implements를 하게 되면 추상클래스를 상속받는 것과 동일한 역할을 하게 되지만 자바스크립트로 컴파일 시 인터페이스와 관련된 코드는 결과물로 나타나지 않게 됩니다. 결과적으로 추상클래스를 사용할 때보다 컴파일 결과물인 코드의 양이 줄어들게 되는 것입니다.

interface implements

인터페이스를 implements하게 되면 해당 클래스의 필드는 반드시 public으로 선언해야합니다. private하게 사용하고자 한다면 추상클래스를 사용해야 합니다.

interface User{
    firstName:string,
    lastName:string,
    greet(name:string):string,
    myName():string,
}

// implements
class Player implements User{
    constructor(
        public firstName:string,
        public lastName:string,
    )

    greet(name:string){
        return `Hello ${name}!`;
    }

    myName(){
        return `My name : ${this.firstName} ${this.lastName}`;
    }
}
